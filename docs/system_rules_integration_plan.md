# 系统规则内置化方案

## 现状分析

### 5大基础规则及问题

| 规则 | 定义 | 当前状态 | 核心问题 |
|------|------|---------|---------|
| **1. Token管理** | 每次对话后检查并报告token消耗，分级提醒 | 规则写在AGENTS.md，但**手动执行**，经常遗漏 | 依赖记忆，非自动化 |
| **2. 记忆和备份** | HEARTBEAT自动执行Git备份、每日记忆创建 | 已配置在HEARTBEAT.md，**自动运行** | 正常 |
| **3. 深度思考** | 复杂任务使用thinking模式 | 按需手动启用 | 需要任务识别机制 |
| **4. 用户画像** | 每次对话读取USER.md，动态更新 | 手动读取和更新 | 需要自动化触发 |
| **5. 持续迭代** | 技能更新、子代理优化 | 被动响应需求 | 需要主动检查机制 |

### 根本原因

规则分散在多个文件中（AGENTS.md、HEARTBEAT.md、SOUL.md），**没有统一的执行引擎**，依赖我"记得"去执行。

---

## 系统化解决方案

### 方案一：创建统一规则引擎（推荐）

创建 `SYSTEM_RULES.md` 作为中央规则库，配合自动化脚本：

```
SYSTEM_RULES.md          # 规则定义
├── rules/
│   ├── token-monitor.json    # Token监控规则
│   ├── memory-backup.json    # 记忆备份规则
│   ├── thinking-mode.json    # 深度思考规则
│   ├── user-profile.json     # 用户画像规则
│   └── iteration-check.json  # 持续迭代规则
└── scripts/
    └── rule-executor.py      # 规则执行引擎
```

### 方案二：利用现有机制增强

1. **Token管理** → 绑定到每次回复的钩子
2. **记忆备份** → 已在HEARTBEAT，保持
3. **深度思考** → 创建任务识别器自动判断
4. **用户画像** → 会话启动时自动加载
5. **持续迭代** → 添加到HEARTBEAT检查项

### 方案三：混合架构（最可行）

结合现有机制和增量改进：

| 规则 | 改进措施 | 自动化程度 |
|------|---------|-----------|
| Token管理 | 每次回复后强制检查 | 100%自动 |
| 记忆备份 | HEARTBEAT保持 | 已100%自动 |
| 深度思考 | 任务复杂度识别器 | 80%自动 |
| 用户画像 | 会话启动自动加载 | 100%自动 |
| 持续迭代 | HEARTBEAT添加检查项 | 80%自动 |

## 具体实施计划

### Phase 1: Token管理自动化（立即执行）

**措施**：
1. 创建 `.rules/token-monitor.json` 规则定义
2. 修改回复流程，强制附加token状态
3. 设置自动提醒触发器

**代码实现**：
```json
{
  "rule": "token-monitor",
  "trigger": "every-response",
  "action": "check-session-status",
  "thresholds": {
    "40%": { "action": "append-hint", "message": "⚡ 上下文已使用 40%+" },
    "50%": { "action": "send-warning", "message": "⚠️ 上下文累计消耗已超过 50%..." },
    "60%": { "action": "send-urgent", "message": "强烈建议新开对话" },
    "70%": { "action": "summarize-and-suggest" }
  }
}
```

### Phase 2: 会话启动自动化

**措施**：
1. 创建会话启动检查清单
2. 自动读取USER.md、SOUL.md、今日记忆
3. 自动加载用户画像

**检查清单**（自动执行）：
```
□ 读取 SOUL.md（人格设定）
□ 读取 USER.md（用户画像）
□ 读取 memory/YYYY-MM-DD.md（今日记忆）
□ 检查多代理系统状态
□ 加载专业技能（如需要）
```

### Phase 3: 深度思考自动识别

**措施**：
创建任务复杂度评估器，自动判断是否需要启用thinking模式：

```
触发thinking模式的关键词/场景：
- "分析"、"设计"、"规划"、"策略"
- 多步骤任务（>3步）
- 需要搜索+整合信息的任务
- 代码编写/系统设计
- 长文档处理
```

### Phase 4: 持续迭代检查

**措施**：添加到HEARTBEAT.md：

```markdown
### 持续迭代检查（每次HEARTBEAT执行）

- [ ] 检查已安装skills是否需要更新
- [ ] 检查子代理运行状态
- [ ] 回顾最近完成的任务，提取可复用模式
- [ ] 检查USER.md是否需要更新（新信息提取）
```
